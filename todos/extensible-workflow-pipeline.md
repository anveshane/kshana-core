# Extensible Workflow Pipeline — Arbitrary Intermediate Processing

## Problem

The executor pipeline is fixed: shot_image → shot_video → final_video. Users can upload custom workflows for video generation, but can't add intermediate processing steps. Workflows that need ControlNet depth maps, SAM masks, pose estimation, style transfer, or any other pre/post-processing are unsupported because the executor doesn't know how to chain workflow outputs.

## What Already Exists

### Designed but not wired:
- **`image_processing` pipeline type** — already in `WorkflowPipeline` enum, manifests can declare it
- **`image_processing` InputSource** — `inputRequirements` can reference `source: 'image_processing'` to consume output from a chained processing step
- **`generateProcessingModesSection()`** — generates LLM prompt section for `{{AVAILABLE_PROCESSING_MODES}}`
- **Scene video prompt guide** — already has `imageProcessingMode` field per shot (LLM can choose a processing workflow)

### What works today:
- User uploads any ComfyUI workflow → manifest created with `parameterMappings`
- `parameterizeGeneric()` applies mappings to any node/field combination
- Strategy-aware routing picks the right video workflow per shot
- Multi-frame image generation (first_frame, last_frame, mid_frame)

## What's Missing

### 1. Executor doesn't run image_processing workflows
The executor has no code path between `shot_image` and `shot_video` to run intermediate processing. When a video workflow declares `source: 'image_processing'` for an input, the executor can't resolve it.

### 2. Chaining outputs between workflows
No mechanism to take the output of one workflow (e.g., depth extraction → depth_map.png) and feed it as input to the next workflow (e.g., ControlNet → video). The resolver needs to:
1. Detect that the video workflow needs an `image_processing` output
2. Find the active image_processing workflow for this shot
3. Run it with the shot_image as input
4. Pass its output to the video workflow

### 3. New input types not generated by existing DAG nodes
Some workflows need inputs that aren't produced by any current node:
- **Depth maps** — from MiDaS, DepthAnything, etc.
- **Edge maps** — from Canny, HED
- **Pose data** — from OpenPose, DWPose
- **Masks** — from SAM, GroundingDINO
- **Style references** — user-provided images

### 4. User-provided inputs at project setup
The `source: 'user'` InputSource exists but the executor doesn't prompt the user for these inputs. Workflows that need a style reference image or a custom mask have no way to collect them.

## Proposed Design

### Phase 1: Image Processing Chain
Wire the existing `image_processing` pipeline into the executor:

1. When a `shot_video` node is about to execute, check if the active video workflow has any `source: 'image_processing'` inputs
2. If yes, find the active image_processing workflow (from `getActiveForPipeline('image_processing')`)
3. Run the image_processing workflow with `shot_image` as input → produces intermediate output (depth map, mask, etc.)
4. Pass the intermediate output to the video workflow via `frameImages` or a new `processingOutputs` field
5. The `imageProcessingMode` field from scene_video_prompt controls which processing workflow to use per shot

### Phase 2: Dynamic Pipeline Stages
Allow manifests to declare arbitrary chaining:

```json
{
  "inputRequirements": [
    { "id": "depth_map", "type": "depth_map", "source": "image_processing", "description": "Depth map from shot image" },
    { "id": "first_frame", "type": "image", "source": "shot_image", "required": true }
  ]
}
```

The executor resolves `source: 'image_processing'` by finding a processing workflow whose `outputType` matches the input's `type` (e.g., depth_map), running it, and passing the result.

### Phase 3: User-Provided Inputs
For `source: 'user'` inputs:
1. During project creation or workflow configuration, collect user-provided files
2. Store them in the project directory
3. Executor resolves `source: 'user'` inputs from stored files

## Example: ControlNet Depth-Guided Video

1. User uploads two workflows:
   - **DepthAnything** (pipeline: `image_processing`, outputType: `depth_map`)
   - **ControlNet LTX** (pipeline: `video_generation`, inputs: `first_frame` + `depth_map` from `image_processing`)
2. LLM generates scene_video_prompt with `imageProcessingMode: "depth_anything"` for shots needing depth guidance
3. Executor runs: shot_image → DepthAnything → depth_map.png → ControlNet LTX (with first_frame + depth_map) → video

## Key Files

- `src/core/planner/ExecutorAgent.ts` — needs image_processing execution step between shot_image and shot_video
- `src/services/providers/WorkflowModeRegistry.ts` — already supports image_processing pipeline, needs chaining logic
- `src/services/providers/comfyui/ComfyUIProvider.ts` — needs to accept processing outputs alongside frame images
- `src/services/providers/types.ts` — `InputSource` already has `image_processing`; may need `processingOutputs` in VideoGenerationInput
- `prompts/skills/defaults/scene_video_prompt_guide.md` — already has `imageProcessingMode` field

## Priority

Medium — extends the system from "custom video workflows" to "custom pipelines". Unblocks ControlNet, inpainting, style transfer, and any future processing step.
